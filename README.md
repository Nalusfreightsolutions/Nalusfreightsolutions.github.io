## What just changed (session 6) — owner login confirmed, RLS recursion bug fixed
1. **Real owner account tested end-to-end for the first time** and hit a bug: after a successful Supabase Auth sign-in, the app fell back to `sales_rep` (only 3 tabs) instead of `owner`. Root cause was a Postgres error, not a role-assignment mistake: `42P17 - infinite recursion detected in policy for relation "profiles"`.
2. **Cause**: the `"owners can read all profiles"` RLS policy on the `profiles` table queried `profiles` from inside its own `USING` clause (`exists (select 1 from profiles p where p.id = auth.uid() and p.role = 'owner')`). Postgres RLS re-triggers the policy check on that inner query too, which recurses forever. The schema already had the correct fix pattern available (a `security definer` function, `is_owner()`) and used it for every other table — it just wasn't applied to this one policy on `profiles` itself.
3. **Fix**: `supabase_schema.sql` updated so `is_owner()` is defined first and the `"owners can read all profiles"` policy now uses `is_owner()` instead of a raw self-referencing subquery. A standalone non-destructive patch, `supabase_hotfix_profiles_rls.sql` (just drops + recreates that one policy, no data touched), was written and run directly against the live DB via the Supabase SQL editor — confirmed fixed by a direct authenticated REST call to `/rest/v1/profiles?select=role&id=eq.<uid>` returning `200 {"role":"owner"}` instead of `500`.
4. **Unrelated local-only snag while testing**: after applying the SQL fix, the Supabase JS client in the specific automated preview browser used for testing got wedged (`sbClient.auth.getSession()` hung indefinitely) — traced to a stale/wedged session token in that browser's `localStorage` after several rapid reloads during debugging, not a real app bug. Clearing `localStorage` and reloading fixed it immediately. Not expected to affect a normal user opening the app fresh in their own browser, but worth knowing if this ever recurs: clear the `sb-nujalvroqdjtbnqzxknq-auth-token` localStorage key and reload.
5. **Confirmed working end-to-end** with the real owner account: full nav (all 11 tabs) visible, user badge shows `nalunui21@gmail.com · Owner`, real data loads (Driver Compliance table populated and correct).
6. **Next step is now unblocked**: run `supabase_temp_open_access_REVERT.sql` to drop the temporary anon-access policies and actually lock the database down at the RLS level (still not done as of end of this session).

## What just changed (session 7, continued) — data-loss safety net
1. **Identified a structural data-loss risk**: every save (`persist()` → `syncStateToSupabase()`) does a full "upsert current in-memory state, then delete anything in the database not present in that in-memory copy" for every table (`deleteMissing()`, [index.html:1096](index.html:1096)), and for child records (contacts/materials on pits/clients/quotes/sales jobs) it's even more aggressive — `replaceChildren()` ([index.html:1106](index.html:1106)) unconditionally deletes *all* children of known parents and reinserts whatever the browser currently has in memory, on every single save. With 5 people now able to use the app, if any one browser tab's local copy is ever incomplete or stale relative to the database (slow network on load, two people editing at once, a tab left open across a schema change, etc.), the very next edit that person makes — anywhere in the app — would silently delete real data. This is exactly what looked like it happened during this session's testing (turned out to be a false alarm, confirmed via the Supabase Table Editor, but the underlying risk is real).
2. **Fix — safety circuit breaker**: `deleteMissing()` and `replaceChildren()` now refuse to proceed if the delete/replace would wipe out more than half of a table (only kicks in above 5 existing rows, to avoid false positives on small tables). Instead of deleting, they throw and show a persistent, dismissible red banner across the top of the app (`#safety-banner`, new element right after `<body>` so it's visible regardless of which tab/screen is active) explaining what was blocked and why, and that the fix is to reload the page for fresh data before making more changes (or, for a rare intentional bulk delete, to do it directly in the Supabase dashboard instead). Verified in the preview: the banner element renders hidden by default and toggles correctly.
3. **Still only a partial answer** — this catches the "silently wipes a large fraction of a table" failure mode, but does not replace real backups. The only backup mechanism today is the manual "💾 Backup Data" button on the Dashboard (downloads the full `state` object as a JSON file — nothing automatic, relies on someone remembering to click it and store the file somewhere safe). **Not yet done**: confirm what Supabase plan/tier this project is on and whether Point-in-Time Recovery or scheduled backups are enabled at the database level — that's the next real safety net to add, independent of anything the app itself does.

## What just changed (session 9) — Pit Map reliability for heavy use

## Resolved — data loss incident (session 14), root cause fixed
**On 2026-07-03, Claude's own live testing deleted all rows in `loads`, `quotes`, and `sales_jobs`.** Owner confirmed afterward that this data was practice/test entries only (client "Phillips" / job "Jarelys" and a handful of loads) — **no real business data was lost.** Real day-to-day use of the app was starting right around when this happened, so this was caught before it could have mattered. The root cause is fixed (see below); leaving the postmortem here since the underlying lesson matters for anyone doing live testing against this app going forward.
- **What happened**: Claude ran test scripts against the live app (filling forms, calling `persist()`) via direct browser console access *before* `loadState()` had finished fetching existing data from Supabase into `state`, and separately via a manual re-invocation of `loadState()`/`persist()` outside the normal boot sequence. Because every table sync in `persist()` works by deleting anything in the database that isn't in the current in-memory list, saving while `state` was still incomplete deleted real rows that just hadn't loaded into memory yet.
- **Why the existing safety net didn't catch it**: the "more than half a table deleted at once" circuit breaker (session 7, `deleteMissing()`/`replaceChildren()` in `index.html`) only engages when `existing.length > 5`. `loads`/`quotes`/`sales_jobs` all had ≤4 rows — under the threshold — so the block never fired even though entire tables got wiped.
- **Fix shipped (session 14)**: added an `initialLoadComplete` flag, set only once `loadState()` has actually finished. `persist()` now refuses to run (logs an error, no-ops) if called before that flag is true. This directly closes the mechanism that caused the incident. Verified live: `persist()` correctly no-ops with a console warning before load completes, and saves normally afterward with no change to normal behavior.
- **Confirmed via direct Supabase queries** that everything else was untouched throughout: `pits` 66, `pit_materials` 457, `pit_contacts` 31, `compliance` 4, `annual_items` 6, `invoice_counters` 2. Only the three practice-data tables were affected.
- **Lesson for future live testing against this app**: any direct-state testing via console/eval must wait for the real, fully-loaded app state (e.g. via the normal sign-in flow, letting `render()` finish) before doing anything that could lead to `persist()` — don't manually re-invoke `loadState()`/`persist()` mid-session outside the normal boot sequence. The new `initialLoadComplete` guard is a backstop, not a reason to be less careful.
- **The safety-net row-count threshold (`> 5`) is still low enough to miss small real tables** — worth reconsidering now that real data entry is starting and tables like `loads`/`quotes` will initially be small. Not changed this session; flagging for later.

Added backward-compatibility for existing loads that were saved under the old flat-total convention: editing an old load now back-calculates an equivalent per-unit rate (`driverPay ÷ qty`) to display in the field, so re-saving it unchanged reproduces the same total instead of silently zeroing it out; the Excel import path got the same fallback for a legacy flat "driverpay" column. Verified all cases (the DHS scenario, unchanged Company Driver behavior with and without a custom rate, NFS, and the backward-compat re-save case) in isolation via JavaScriptCore before shipping — every number matched exactly.
